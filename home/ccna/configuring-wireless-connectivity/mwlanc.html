Manage WLAN Connections

One of the main responsibilities of a CCNA is to manage WLAN connections. This topic describes the options you have for connecting to WLCs and APs so that you can manage them.

Console Port

When you use the CLI to configure Cisco network devices, you have the option of using a direct serial connection to the console port, or you can use a remote network session. You typically connect to the AP's console port to configure it to be either stand-alone, with its own configuration, or to be a LAP controlled by a WLC.

To directly connect a computer to a Cisco device such as a WLC, you need:

  • A Cisco console cable to plug into the controller's console port and the serial port of your computer. One end of the cable has an RJ-45 connector for the console port, and the other end has a serial connector for the serial port.
  • A computer that must be running a terminal emulation program such as PuTTY. This program should be configured with default settings.
  • A valid user name and password for logging on to the device.

Telnet

Telnet is a protocol that enables TCP/IP connections to a host. Telnet enables a user at one site to establish a TCP connection to a login server at another site and then allows keystrokes to be passed from one device to the other. Telnet can accept either an IP address or a domain name as the address of the remote device. The Telnet application protocol invokes the terminal emulator or a Telnet client. The Telnet server is the device that listens for the commands and replies to the commands. Because Telnet sends all traffic in clear text, it is not a secure form of management and SSH should be used instead whenever possible.

For managing Cisco devices remotely using Telnet, you need:

  • A computer that is connected to the network and that can reach the device's management IP address or service port address.
  • The IP address of the device.
  • A VT-100 terminal emulation program to run the Telnet session. The program should be configured with an IP address and to use TCP port 23.
  • A valid user name and password to log in to the device.

In addition, the device needs to be configured to accept Telnet connections.

SSH

Secure Shell (SSH) is a protocol that provides a secure remote access connection to network devices. Unlike Telnet, both versions of SSH (1 and 2) encrypt the communications between the client and server. You should implement SSH version 2 whenever possible because it uses a more enhanced security encryption algorithm than SSH version 1.

To enable SSH support on a Cisco WLC, you would perform the following process:

1. Configure the hostname command to specify a valid host name.
2. Configure the Domain Name System (DNS) domain to define your domain.
3. Generate the SSH key to be used.
4. Enable SSH transport support for the virtual type terminal.
  • Note: On Windows clients, you will need to download an SSH client application such as PuTTY.

Command Reference: Configuring SSH

This section contains the commands you will use in the next lab.

Configuring SSH

The following commands are used in the Configuring SSH lab:

  • configure terminal: Enter global configuration mode from privileged EXEC mode.
  • crypto key generate rsa: Generate RSA key pairs.
  • enable: Enter privileged EXEC mode.
  • end: Ends and exits configuration mode.
  • exit: Exits one level in the menu structure.
  • ip domain-name <name>: Defines a default domain name to complete unqualified host names.
  • ip ssh time-out <seconds>: Configures the time interval that the router waits for the SSH client to respond.
  • ip ssh version [1 | 2]: Specifies the version of SSH to be run on a device.
  • line vty 0 4: Enters configuration mode for the specified virtual terminal (VTY) lines.
  • login local: Configures the device to use its local database for authentication at login.
  • ping <ip-address>: Sends an Internet Control Message Protocol (ICMP) echo request to the specified address.
  • show ip ssh: Displays the version and configuration data for SSH.
  • show running-config: Displays the active configuration file.
  • show ssh: Displays the status of SSH server connections on the device.
  • ssl [-l <user-id>] <ip-address> | <host-name>: Starts an encrypted session with a remote networking device.
  • telnet <ip-address>: Starts the terminal emulation program from a PC, router, or switch - permits you to access devices remotely over the network.
  • transport input [all | none | ssh | telnet]: Defines which protocols to connect to a specific line of a router.
  • username <name> privilege <privilege-level> password <password>: Creates a local user name and password pair and sets the privilege level for the user.

HTTP and HTTPS

If you want to use a graphical interface to manage your WLANs, you can enable the controller's distribution system port as a web port or secure web port by using HyperText Transfer Protocol (HTTP) and HyperText Transfer Protocol Secure (HTTPS), respectively.

  • HTTP is a protocol that forms the basis of the World Wide Web. It defines how messages are formatted and transmitted, along with how web servers and browsers should respond to various commands.
  • HTTPS is an extension of HTTP that uses Secure Sockets Layer (SSL) or Transport Layer Security (TLS) to encrypt communications. Once HTTPS is enabled, the controller generates a local web administration SSL certificate and applies it to the management GUI.

You can use the GUI or CLI to enable web and secure web modes.

TACACS+

You can use TACACS+ to provide authentication services to enterprise WLANs. Terminal Access Controller Access Control System Plus (TACACS+) is a Cisco proprietary protocol that enables centralized authentication of remote user access to network devices through authentication, authorization, and accounting (AAA). Implementing TACACS+ requires a TACACS+ server. A TACACS+ daemon running on the server supports a database that stores TACACS+ service information. It enables the use of a single access control server with separate databases to handle AAA services independently. It uses TCP port 49 and supports multifactor authentication.

Radius

Remote Authentication Dial-In User Service (RADIUS) is an open standard that enables a server to provide standardized and centralized authentication for remote users through AAA. RADIUS uses UDP port numbers 1812 and 1813. You can configure an authentication server as a RADIUS server, and all of the other devices that access the server, including remote access servers, can be configured as RADIUS clients. The RADIUS clients pass all authentication requests to the RADIUS server. You can configure Cisco routers, APs, and switches as RADIUS clients. User configuration, remote access policies, and usage logging are centralized on the RADIUS server.

RADIUS is based on a distributed client/server architecture. It provides for centralized authentication in a network that includes servers from different vendors. It can also be used on networks where user access must be restricted to a specific service, resource accoutning services must be provided, or users must be authenticated to access specific resources. RADIUS does not support two-way authentication and is not suitable for network environments that use multiple protocols. RADIUS associates users with one particular service model. Therefore, it is not suitable for networks that need to support multiple services.
