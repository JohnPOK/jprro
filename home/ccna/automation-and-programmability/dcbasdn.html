Describe Controller-Based and Software Defined Networking

Topic B

Controller-Based Networking and Software-Defined Networking

One of the most efficient ways to use automation in network management is to use software to perform certain management tasks.

Traditional Networks and Software-Defined Networking

Traditional networks are based on physical hardware devices and are generally referred to as static networks. Their flexibility is limited, as any change must be manually implemented. Software-defined networking (SDN) is (as its name implies) based on software and provides much more flexibility than traditional networks.

The key characteristic of SDN is that it controls network routers and switches through the use of open-source software on generic switches and routers.

Controller-Based Networking

Controller-based networking is a popular implementation of SDN. You have already seen that controllers are often implemented in a WLAN, but this is not the only solution they can provide for your networks:

  • Controller-based networking provides administrators with increased control over the flow of network traffic. When the need arises, the administrator can change switching rules by changing information on the controller, which then communicates the changes to the network devices.
  • Controller-based networking enables administrators to use the controller to distribute changes without having to configure individual devices.
  • The traffic monitoring capabilities of a controller can help you identify suspicious network traffic and deal with it accordingly.
  • Service virtualization reduces your network's reliance on expensive hardware.

Architectures

You should recall the three-tier architecture and collapsed core architectures that were introduced in the switching discussions. For network management, the control/data plane architecture might be the most recognized. In this architecture, there are actually three layers, or planes: the application plane, the control/management plane, and the data forwarding/operational plane.

  • The application plane contains applications and services that make requests for network functions from the control plane and the data plane.
  • The control plane (CP) carries signaling traffic and performs routing functions, as well as system configuration and management.
    • Note: Some references cite the management plane as one that carries administrative traffic and that it is considered a subset of the control plane.
  • The data forwarding/operational plane carries the traffic sent to and received by network users.

Unlike traditional networking, where all three planes are incorporated into the firmware of routers and switches, SDN separates the control (and management) plane from the hardware and replaces it with software. This has the distinct advantage of supporting centralized administration of network devices.

For SDN, the architectural components include:

  • SDN applications, which communicate their network requirements to an SDN controller.
  • SDN controllers, which interpret the requirements of SDN applications and are considered logically centralized components. This does not mean that you can have only one SDN controller - in fact, the implementation of multiple controllers (whether in a peer or hierarchical configuration) is quite common.
  • SDN datapaths, which are logical network devices that emulate a network switch or other device.
  • The SDN Control to Data-Plane Interface (CDPI), which is the interface defined between an SDN controller and an SDN datapath and that provides programmatic control of all forwarding operations, capabilities advertisement, statistics reporting, and event notification.

Overlay and Underlay Networks

An overlay network is a virtual network that is built on top of another network and connected by virtual links. Most SDN implementations are carried out on existing networks, so the existing switches and routers remain in place, and the SDN technology is linked to it to provide improved flexibility and manageability.

An underlay network is simply the physical infrastructure that the overlay network is linked to.

Fabric

Fabric is a network topology where nodes communicate through one or more interconnected network switches. The interconnected switches often form a mesh network that can provide redundancy and reliability.

When you combine SDN with the use of fabric, there are two basic options for implementation:

  • Obtain SDN software and network fabric from different vendors, and configure SDN to control the fabric.
  • Obtain network fabric that has SDN technology integrated into it. This fabric is more application-aware and can support QoS.

Control Plane and Data Plane

The control and data planes are integral components of the SDN architecture. Protocols and network device functions help to separate the data plane and control plane on each network device.

Type of Plane Description
Control plane
  • A collection of actions that control the data plane.
  • Provides information that the data plane requires to function, such as IP routes and MAC address entries.
  • Includes operations for creating and managing routing tables, MAC address tables, and ARP tables.
  • OSPF is a routing protocol that manages the routing tables in the control plane on each router.
  • The control plane enables the data plane to work efficiently.
  • The most commonly used protocols in the control plane are IPv4 ARP, IPv6 Neighbor Discovery Protocol (NDP), STP, and MAC Address Learning in switches, and routing protocols such as OSPF, RIP, BGP, and EIGRP.
Data plane
  • A collection of tasks performed by a networking device when it handles data.
  • Includes operations performed by the device when it receives, processes, and transmits data (frames, packets, or messages).
  • One of the most important tasks of a router or a switch in the data plane is forwarding data.
  • The data plane on each router uses the routing tables to forward packets.
  • The data plane is also called the forwarding plane.

Network Devices and Data Plane Actions

This table lists some devices and features, and identifies the actions they perform in the data plane.

Network Device or Feature Performs Data Plane Actions To:
Routers and Layer 2 switches Add or remove 802.1Q headers for VLAN trunks.
Routers and Layer 3 switches De-encapsulate and then encapsulate packets into frames.
Layer 3 switches Match the IP address of the destination to the routing table.
Layer 2 switches Match the MAC address of the destination to the MAC address table.
ACLs and security Filter packets and drop messages that do not meet the specified ACL criteria or security levels.
NAT processing Translate the IP address of the source or destination.
VPN processing Encrypt data and add IP headers for VPN processing.

Management Plane

The management plane is a subset of the control plane.

  • It is a collection of protocols that enable management of network devices.
  • Management plane protocols are not essential for the working of networking devices.
  • Some of the most commonly used protocols in the management plane are Telnet, SSH, SNMP, and Syslog.
  • Protocols such as SNMP and Syslog in the management plane help to troubleshoot and diagnose problems, perform maintenance tasks, and monitor device performance.

Application Programming Interfaces (APIs)

In general computing terms, an application programming interface (API) is a collection of commands, functions, protocols, and objects that programmers use to create software or interact with other computers or devices without the need for creating all the code from scratch. They are often incorporated into software development kits (SDKs).

When it comes to SDN, APIs enable the exchange of data between a controller and the programs running on network devices. These are generally referred to as northbound APIs and southbound APIs.

  • The northbound APIs enable applications to communicate with controllers.
  • The southbound APIs enable controllers to communicate with switches and routers.

Northbound and Southbound APIs

Northbound and southbound APIs are named as such because in most network diagrams, controllers are found above (or north of) the routers and switches they manage.

A southbound interface (SBI) enables communication between a controller and a network device. The SBI is configured with the protocols and the southbound API needed to establish a link between the controller and a network device. The controller uses the southbound API to pass on routing tables, MAC address tables, and other control information to the network devices.

To perform its functions, a controller stores several pieces of information such as a list of all devices in the network, the network topology, interfaces on each device, device configurations (including IP addresses and VLANs), device capabilities, and current state of devices.

A northbound interface (NBI) is an interface that enables a program or an application to obtain the information stored on a controller and to process that information. The northbound API enables the exchange of data between the application and the controller. By using the information obtained from the controller, the application programs the logic that the controller uses to maintain forwarding tables for the network.
