Topic B

Manage passwords

Many AAA implementations require a valid user name and password for authentication. As a CCNA, you will likely be responsible for managing passwords.

Password Policies

A security policy is a formalized statement that defines how security will be implemented within an organization. It describes the means the organization will take to protect the confidentiality, integrity, and availability of sensitive data and resources. A password policy is a subset of an overall security policy that documents standards for creating, storing, and modifying passwords.

Password policies define criteria for the use of local passwords, general password management, and password complexity, among other things.

Cisco Authentication

Cisco devices have several options for authentication. These methods are applicable whether you are making a local connection at the console or a remote connection by using Telnet or SSH.

  • Authenticate to the local device with a password only.
  • Authenticate to the local device with a user name and password combination.
  • Have your authentication forwarded to a TACACS+ or RADIUS server.
  • Once logged in, elevate your privilege level with an enable password or enable secret password.

Providing a user name along with the password allows the administrator to automatically grant the user a particular privilege level when they log in. If the user logs in with only a password, they enter at User EXEC (unprivileged) mode. If they want to then escalate to privileged mode, they then must type enable and provide the enable password. If the administrator wants the user to automatically enter in privileged mode (without having to type the enable password), they set the privilege level while creating the user account. For example, username suzy privilege 15 password letmein will create the user suzy with the password of letmein. When Suzy logs in, she is automatically at privileged EXEC mode and does not have to enter the enable password.

The enable command can be given a password to protect privileged EXEC mode. This is not required, but is best practice. Historically, the enable password was stored in plain text or by using weak encryption that can be easily broken. To address this weakness, Cisco added the enable secret password, which is stored as an MD5 hash. You can configure both the enable password and enable secret password on the same device. However, if you do, then the device will use only the enable secret password. A user cannot specify which version they wish to use. The enable password is used only if the administrator deletes the enable secret password from the device configuration.

1. Local authentication: User names and passwords stored in the device's running-config (and startup-config) file.
2. Remote authentication: User names and passwords stored on a RADIUS or TACACS+ server or a separate Directory Services server.
3. Logging in with password only or as a user with privilege level 1 displays the user EXEC prompt.
4. Enable password is stored on the device in plain text or with weak encryption, and enable secret password is stored on the device with strong MD5 encryption. If you are at the user EXEC prompt, you must provide either the enable password or enable secret password to gain a privileged EXEC prompt.
5. Logging in as a user with privilege level 15 displays the privileged EXEC prompt immediately.

Password Encryption Levels

Cisco offers several levels of password encryption, as described in the following table.

Password Encryption Level Description
Type 0 There is no encryption. The password is stored as clear text. This level should not be used.
Type 4 An older level failed to provide strong password hashes. This level should not be used. It was deprecated with the relaese of IOS 15.3(3).
Type 5 This type uses MD5 encryption. The encryption is crackable but it should take a long time to crack it. Use this level only when type 8 or 9 is not available in your version of IOS.
Type 7 This type uses the Vigenere cipher, which is easily cracked in a matter of seconds or faster. This level should not be used.
  • Note: This is the level that IOS applies when you issue the service password-encryption command.
Type 8 This type uses Password-Based Key Deriviation Function 2 (PBKDF2) with Secure Hash Algorithm, 26-bits (SHA-256) as the hashing algorithm.
Type 9 This type uses scrypt as the hashing algorithm [starting from IOS version 15.3(3)]. If your IOS version supports it, it is by far the hardest to crack.

Local Passwords

On a Cisco networking device, a local password is one that is stored in the device configuration. Some devices, such as the ASA firewall, maintain a separate local database. Otherwise, local user names and passwords are simply stored in the device startup-config and running-config files. Depending on the device, purpose, and IOS version, they might be stored in plain text or as a hash. Later versions of the IOS included commands to encrypt any existing plain text passwords in running-config.

Password Management

With all the different types and levels of passwords on a Cisco network, it can be a daunting task to manage them. For remote authentication, you can manage user names and passwords at the AAA server or Directory Services server. Management systems such as Cisco DNA can also be configured with global or individual device user names and passwords, SNMP community strings, authentication types, enable/enable secret passwords, and other authentication information needed to discover and manage devices on the network.

Password Complexity

You can specify a level of password complexity that will make it more difficult for passwords to be cracked. Most organizations require:

  • Minimum password length, such as 7 characters.
  • Allowable characters, such as upper and lower case letters, numbers, and symbols.
  • Required characters, such as at least one upper case letter and at least one symbol.
  • Prohibitions, such as no words found in dictionaries, and no consecutive or repeated characters.

Alternatives to Using Passwords

There are other authentication methods that do not use passwords as the base or sole authentication mechanism. Some of these include:

  • Multifactor authentication
  • Biometrics
  • Digital certificate

Multifactor Authentication

Authentication factors are traditionally listed as:

  • Something you know, such as a password or PIN.
  • Something you have, such as a token, smart card, or swipe card.
  • Something you are, which relates to physical characteristics such as fingerprints or retinal patterns.

Some newer factors include:

  • Where you are, such as directly connected to the network or remotely connected.
  • What you do, such as keystroke patterns.

For multifactor authentication (MFA) to be in place, you need to include at least two distinct authentication factors. For instance, if you have a user name and password (what you know) and a PIN (what you know), this is not multifactor authentication. However, if you have a user name and password (what you know) and a key fob (what you have) that generates one-time passwords, this is a valid example of multifactor authentication.

Some commonly used MFA schemes include:

  • Using a one-time password (OTP) to gain access to a secure website such as a bank's website. Generally speaking, you enter your user name and password (what you know) credentials on a webpage, and the website sends you a time-sensitive message via email or SMS text that contains an authorization code (what you have) for you to access the website's resources.
  • An employee ID badge (what you have) and a fingerprint or retinal scanner (who you are) used to gain entry to a secure portion of a building.

Biometrics

Biometrics is the measurement and analysis of physical characteristics such as fingerprints, voice patterns, and retinal patterns for use in proving an individual's identity. Many keyboards and mobile phones employ biometric technology to assist in protecting equipment and data from unauthorized access. Facial recognition software is also an option.

Digital Certificates

A digital certificate is an electronic document that is used to identify a person, device, organization, or some other entity. You can use digital certificates for device authentication as well as user authentication. You can also combine certificates with a user name/password combination to enhance security.

There are two main categories of digital certificates: certificates signed by a certificate authority (CA) and self-signed certificates. A self-signed certificate is a digital certificate that does not have a signature from a CA. Generally speaking, certificates signed by a CA can provide more security, but there are times when using a self-signed certificate should be considered.

Because self-signed certificates are easy to create and incur no costs, they should be used on internal LANs without making the network and its data more prone to attack. However, if you do use self-signed certificates, there is a chance that your users will receive a warning that the certificate is not trusted. In this case, they would have to acknowledge the warning to continue their tasks.

For networks that have WAN links, it is strongly recommended that CA-issued certificates be used.
