Troubleshoot Network Management Issues

As with other facets of network operation, part of the administrator's tasks involve troubleshooting issues that prevent or hinder network management.

Common Network Management Issues

In addition to physical and underlying network issues, some common network management issues include:

  • Unsynchronized clocks.
  • Misconfigured SNMP community string.
  • Hosts too busy to respond to SNMP.
  • Hosts and devices not configured to use Telnet or SSH.
  • Access control and packet filtering restricting Telnet, SSH, SNMP, and ICMP traffic.

Guidelines for Troubleshooting Network Management Issues

Here are some guidelines for troubleshooting network management issues:

Troubleshooting SNMP Connectivity Issues

Consider the following when you encounter issues with SNMP connectivity:

  • Verify that the SNMP agent services are running on each device.
  • Verify that the SNMP community string is correctly configured on each device.
  • Verify that UDP ports 161 and 162 are not blocked.
Troubleshooting Telnet Access to a Device

Some of the factors to consider when you are troubleshooting Telnet access to a device include:

  • Verify that the IP address of the remote router or switch is reachable. You can use the ping command to test this.
  • Verify that the correct transport protocols are defined correctly. By default, with IOS 15.0 and later, Telnet and SSH are permitted, and if other protocols are supported, they are typically permitted as well. However, when you use the transport input command, you can change the transport protocols that are allowed. You can verify the allowed protocols with the show line vty line-number | include allowed command.
  • Determine if telnet is configured to ask the user for credentials. By default, it will be configured to do so.
    • The login command prompts the user for a password. However, when you need to authenticate the user through the local database, you need to use the login local command, and if you need to authenticate the user via AAA, you need to use the login authentication { default | list-name } command.
    • Determine if a password is specified. Because the login command is enabled by default, you need to specify a password. If you have not set the password, the error message Password required, but none set will be displayed. When you are using the login local command or AAA, you are prompted to enter a user name and password instead.
  • Determine if there is an ACL denying access for the IP address that can access the router or switch. To receive a log message indicating the IP address that was denied, you may need to add the log keyword to the end of the explicit deny entry in the ACL. If the log keyword is added, the log message %SEC-6-IPACCESSLOGS: list 1 denied <ip-address> packet is displayed.
  • Check to see if all vty lines are busy. Remember that by default, there are five vty lines on Cisco routers and switches, which are numbered 0 to 4. Some devices may have more vty lines. However, regardless of the number you specify, when all the lines have established connections, a new connection will not be established. The next device that tries to telnet will be refused and receive the message Password required, but none set, even though that is not technically the actual issue. You need to use the clear line command, if you need to manually clear the lines, and specify the line numbers that you need to clear before vty. Do not use the actual vty number that is listed after vty.
  • Determine if there is an ACL in the path between the client and the device that is blocking port 23. Remember that Telnet uses TCP port 23. When there is an ACL configured on a router or firewall that blocks port 23, you will be unable to establish a successful Telnet connection.
Troubleshooting SSH Access to a Device

With SSH, you can experience the same issues as with Telnet. Further, SSH has some additional issues:

  • Verify that the correct version of SSH is specified. By default versions 1 and 2 are enabled. However, you can use the ip ssh version { 1 | 2 } command to change it to just 1 or 2. When clients are connecting with v2 and you have configured the device for v1, the SSH connection will fail. This is also applicable when the clients are using v1 and you have configured devices for v2.
  • You need to use the show ip ssh command to view the version of SSH that is running. If the version is displayed as version 1.99, it indicates that both version 1 and 2 are running. When version 1 is displayed in the output, then it indicates that only SSHv1 is running, and when version 2 is displayed in the output, then it indicates that only SSHv2 is running.
  • Verify that the correct login command has been specified. SSH uses a user name and password for authentication and the login command will not work in situations where only the password is requested. The login local command enables you to authenticate with the local database, and the login authentication {default | <list-name>} command enables you to authenticate with an AAA server.
  • Verify that the correct key size has been specified. Remember that SSHv2 uses an RSA key size of 768 or above. When you have been using a smaller key size with SSHv1 and then switched over to SSHv2, you need to create a new key with the correct size. Otherwise, SSHv2 will not work. When you are using SSHv2 but accidentally specify a key size less than 768, you will not be allowed to use SSHv2 connections.
  • Determine if there is an ACL in the path between the client and the device that is blocking port 22. Remember that SSH uses TCP port 22. When you configure an ACL to block port 22 on a router or firewall, you will be unable to establish a successful SSH connection.
