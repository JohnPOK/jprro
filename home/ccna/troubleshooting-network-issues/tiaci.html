Troubleshooting Interface and Cable Issues

When physical layer components fail, it is very likely that the network connectivity will be affected, whether it be one user or a building full of users.

Common Interface and Cable Issues

Common interface and cable issues include:

  • Bad or damaged cable. Generally speaking, if only one computer or device is affected, the problem could be a bad or damaged cable. Some of the possible symptoms are lost connections, slow performance over the network, or intermittent connectivity. You can easily swap a suspect cable with an Ethernet cable that you know is good. If the problems disappear, your troubleshooting is complete.
  • Auto-negotiation and NIC issues. Sometimes there can be problems between Cisco switches and certain third-party NICs. Remember that by default, Catalyst switch ports and interfaces are set to auto-negotiate. Also, it is common for devices like laptops or other devices to be set to auto-negotiate as well. However, sometimes auto-negotiation issues may still occur. To troubleshoot auto-negotiation problems, it is recommended that you try hardcoding both sides. If neither auto-negotiation or hardcoding appears to work, there may be a problem with the firmware or software on your NIC. You may need to upgrade the NIC driver to the latest version available on the website of the manufacturer to resolve this issue.
  • Ethernet and fiber cables. You need to ensure that you are using the correct cable type for the connections. For Ethernet, you need to use Category 3 copper cable for 10 Mbps UTP connections. You need to use Category 5 UTP copper cable for 10/100 Mbps connections, Category 5e or Category 6 UTP copper cable for 10/100/1000 Mbps connections, and Category 6A cable for cameras and WAPs. For fiber-optic connections, you need to verify that you have the correct cable for the distance and port type. You need to ensure that the connected device ports both match and use the same type encoding, optical frequency, and fiber type. For copper connections, you need to determine if a crossover cable was used when a straight-through was required or vice-versa. You may have to enable Auto MDI-X on the switch, or replace the cable.

The types of errors related to interfaces and cable are:

  • Input queue drops: This indicates that a router received information faster than it could process the information.
  • Output queue drops: This indicates that a router is not able to send information out through the outgoing interface due to congestion or possibly due to an input/output speed mismatch.
  • Input errors: This indicates that frames were not received correctly. For example, a cyclic redundancy check (CRC) error may have occured, which could indicate that a cabling problem or a duplex mismatch has occured.
  • Output errors: This indicates that the frames were not transmitted correctly, possibly due to a duplex mismatch.

Physical Damage

You should always check cables for physical damage or failure. A cable may be just good enough to connect at the Physical layer, but it might corrupt packets as a result of subtle damage to the wiring or connectors. You need to check or swap the copper or fiber cable. Also, you may need to swap the transceiver, whether it is a Gigabit Interface Converter (GBIC), SFP or SFP+ transceiver, if removable, for fiber connections.

You also need to verify and rule out any bad patch panel connections or media convertors between source and destination. You may need to try the cable in another port or interface if one is available and see if the problem continues.

Be aware that visual inspection of the media can go only so far. To check for cable breaks that cannot be detected visually, you will need to use a TDR or other cable-testing tool.

Speed or Duplex Mismatch

Duplex mode can be of two types: full-duplex and half-duplex. Full-duplex mode enables devices to send and receive data at the same time. Half-duplex mode allows the devices to perform only one action, either sending data or receiving data, at the same time. Both duplex and speed settings are interrelated because the speed of transmission specified for an interface will affect its duplex settings.

The common issue with speed or duplex occurs when the duplex settings don't match between two switches, between a switch and a router, or between the switch and a workstation or server. This can happen when you are manually hardcoding the speed and duplex settings, or from auto-negotiation issues between the two devices.

While there may be other causes for late collisions, duplex mismatch is probably the most common reason. You can either set both sides of the connection to auto-negotiate speed/duplex, or you can set the speed/duplex manually on both sides. Setting auto speed/duplex on one side and 100/Full-duplex on the other side is also a misconfiguration, and can result in a duplex mismatch. When the switch port receives a lot of late collisions, this usually indicates that there is a duplex mismatch problem and can result in the port being placed in an errdisable state. The half duplex side expects packets only at certain times. If it receives packets at any time other than the expected times, it will consider such packets as collisions.

Collisions

A collision refers to a situation in the network when two nodes transmit data simultaneously without being aware that the other node is also sending data at the same time. This occurs when two nodes listen for network traffic, "hear" none or detect no traffic on the network, and then transmit data simultaneously. Collisions are normal for interfaces configured as half-duplex but should not be seen on full-duplex interfaces. If collisions increase dramatically, this points to a highly utilized link or possibly a duplex mismatch with the attached device.

The show interface ethernet command will show you the number of times a collision occurred before the interface transmitted a frame to the media successfully. The command output shows statistics for collisions and late collisions.

A late collision occurs when two devices transmit at the same time, and neither side of the connection detects a collision. The reason for this occurrence is because the time to propagate the signal from one end of the network to another is longer than the time to put the entire packet on the network. The two devices that cause the late collision never see that the other is sending until after it puts the entire packet on the network. Late collisions are not detected by the transmitter until after the first 64 byte slot time. This is because they are only detected in transmissions of packets longer than 64 bytes.

The excessive collisions counter increases after 16 consecutive late collisions have occurred. After 16 attempts have been made to send the packet, the packet is dropped and the excessive collisions counter increments. If this counter increment occurs, it is an indication of a wiring problem, an excessively loaded network, or a duplex mismatch. An excessively loaded network can be caused by too many devices on a shared Ethernet.

You can use the show controller ethernet <interface-number> command to see the number of excessive collisions.

Guidelines for Troubleshooting Interface and Cable Issues

Here are some guidelines for troubleshooting interface and cable issues.

Troubleshooting Interface and Cable Issues

The following table lists symptoms, causes, and possible solutions for some common interface and cable issues.

Issue and Symptom/Cause Solutions
Excessive noise--When you view the output of the show interface ethernet command, it contains more CRC errors but fewer collisions, which is an indicator of excessive noise on the interface.
  • Check for damage to any of the cables.
  • Check for signal reflections due to any badly spaced Test Access Points (TAPs), which are hardware devices that enable you to access data that is sent on a network.
  • Ensure that for 100BaseTX medium, you are using only Category 5 cabling and not any other type like Category 3 cabling.
Excessive collisions--Slow Network performance.
  • When you view the output of the show interface ethernet command, it shows the number of collisions with respect to the total number of output packets.
  • Use a Time-Domain Reflectometer (TDR) to find any unterminated Ethernet cables. A TDR is an electronic instrument that can be used to characterize and locate faults in metallic cables, such as twisted wire pairs and coaxial cables.
  • Look for any transceiver attached to a host that is constantly sending unwanted signals, which can disrupt the communication within the network. You may need to perform a host-by-host inspection or use a protocol analyzer.
Late collisions--Late collisions will normally not occur in a properly designed Ethernet network. They will usually occur when the Ethernet cables used are too long or when too many repeaters are used in the network.
  • Check for late collisions by using a protocol analyzer.
  • Check the diameter of the network, and ensure that it is within specification.
Excessive runt frames--When runt frames occur in a shared Ethernet environment but the collision rate is not high or they occur in a switched Ethernet environment, they may be due to underruns or bad software on a NIC.
  • A runt frame is less then 64 bytes in size and has a bad CRC. When runt frames occur in a shared Ethernet environment, they are mostly caused by collisions. If this is the case, you need to apply the same solutions given for excessive collisions.
  • Use a protocol analyzer to try to determine the source address of the runt frames.
No link integrity on 10BaseT, 100BaseT4, or 100BaseTX--Wrong wire type being used, mismatch in card, cross-connect of cables, or excessive noise may cause the issue.
  • Make sure you are not using 100BaseT4 when only two pairs of wire are available, because 100BaseT4 requires four pairs.
  • Check for a mismatch between 10BaseT, 100BaseT4, or 100BaseTX. This may help to detect if any card that is different from the switch port is being used.
  • Identify whether there is cross-connect. For example, ensure that only straight-through cables are used between a station and the switch. Cross-over cables must not be used between a station and the switch.
  • Identify any excessive noise.
Troubleshooting Speed and Duplex Issues

When you are attempting to troubleshoot issues related to speed and duplex, you need to look at both endpoints of the link. This will help you verify that the devices are using the same speed and duplex settings. If the endpoints are not using the same speed and duplex setting, you might experience the following issues.

  • Mismatch in Speed: When the transmission speed of the endpoints are not the same, both endpoints will show the interface status as not connected or inactive.
  • Mismatch in Duplex: When the duplex settings of the endpoints are not the same, the endpoint that has the half-duplex setting specified will show problems and will not be able to communicate properly with the endpoint that has full-duplex settings specified, although the interfaces will appear to be active.
