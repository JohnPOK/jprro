Topic G

Troubleshoot Network Services Issues

You also need to be able to identify and correct problems that prevent user from accessing network services such as DHCP and DNS.

Common Network Services Issues

Here are some common network services issues that you might encounter on your network:

Name resolution and DNS issues. There are three possible causes for a client being unable to resolve names to IP addresses or to connect to a DNS server.
  • The client might have an incorrect DNS server setting. This can happen when configuration parameters are entered manually, or when a DHCP server provides the wrong address to the client.
  • There is a problem in establishing IP connectivity between the user's host and the DNS server.
  • The client is not using DNS at all. Its DNS resolver cache is corrupt, or it found an (incorrect) name-to-IP mapping in its HOSTS file and is sending traffic to the wrong IP address.
DHCP configuration issues
  • Before configuring DHCP, you need to consider some key points:
    • DHCP clients do not have an IP address configured initially. They must send a broadcast request to obtain an IP address from a DHCP server.
    • By default, routers do not forward broadcasts. DHCP broadcast requests must be allowed by the routers, when the DHCP server is on another broadcast domain on the Layer 3 network. This task is performed by using a DHCP relay agent.
    • The interface-level ip helper command provide the implementation of DHCP relay by the Cisco router.
DHCP issues. The possible causes of DHCP problems include:
  • Incompatibility between the default configuration of IOS device and configuration of DHCP/BOOTstrap Protocol (BOOTP) Relay Agent, compatibility issue with NIC or issue with a DHCP feature, faulty NIC or improper installation of NIC driver, frequent spanning tree computations causing intermittent network outages, version of operating system or defect in software, misconfiguration of DHCP server scope, and defect in IOS DHCP/BootP Relay Agent software.
  • The failure of the Unicast Reverse Path Forwarding (uRPF) check because the DHCP offer is received on an interface that is different from the expected interface.
    • Note: When you enable the Reverse Path Forwarding (RPF) feature on an interface, a Cisco router may drop DHCP and BOOTstrap Protocol (BOOTP) packets with source addresses of 0.0.0.0 and destination addresses of 255.255.255.255. The router may also drop all IP packets that have multicast IP destinations at the interface.
  • DHCP conflict logging being enabled, even when the DHCP database agent is not used.
NAT issues
  • Issues with source NAT. Most of the issues with source NAT are related to mistakes in the NAT configuration.
    • There may be an issue with the configuration of the interface. Ensure that the interfaces that will be the outside and inside interfaces are properly specified. Otherwise, NAT will not be able to translate the addresses properly.
    • There may be an issue with the configuration of the pool. The pool, which represents the inside global addresses, must contain the correct public addresses that will need to be translated.
    • There may be an issue in reaching the public addresses in the pool. If the public address is not reachable or advertised to the Internet, the traffic will be unable to return from the destination.
    • The correct inside devices may not be referenced by the access list. NAT uses the inside local addresses identified by the ACL for translation to inside global addresses. If the ACL has any incorrect values, the addresses may not be translated correctly.
    • The ACL and NAT pool might not be mapped correctly. The ip nat inside source command allows you to create a mapping between the ACL and the pool.
    • The overload keyword may be missing. You need to include the overload keyword in the ip nat inside source command in order to enable PAT.
  • NAT-related issues for end users. NAT enables you to resolve connectivity issues to the Internet. However, you need to be aware of the potential issues that NAT can create for end users.
    • NAT modifies the checksum for packets because it changes the information about the IP address. However, some VPN protocols verify the packet's integrity by using the packet's checksum. So, it is possible that a packet whose checksum has been modified by NAT could be rejected by a VPN protocol such as IPsec (because the packet might appear to have been altered).
    • NAT hides the true IP address information, because it translates an inside IP address to an outside IP address. This makes it challenging to trace a data flow from end to end for the purpose of troubleshooting. Use the show ip nat translation command to verify that the translation exists in the translation table.
    • Some applications are not compatible with NAT. This is because, when they are being initialized, some applications randomly determine the ports that are to be used for communication. This feature might be incompatible with how incoming traffic is handled by NAT.
    • NAT modifies the Layer 3 information in packets, so the packets may be subject to a slightly longer delay than others. When routers perform numerous NAT translations, this delay can be more evident.
NTP issues
  • Some applications are time-sensitive and require that the system time on all relevant computers, servers, and network devices be synchronised. Normally the implementation of NTP servers addresses this issue, but be aware of these possible problems that could arise on your network:
    • Some Cisco routers might not be able to synchronize if internal time servers are Active Directory objects. The routers use NTP, while Active Directory domains use the Simple Network Time Protocol (SNTP).
    • Routers might not be able to synchronize with public time servers. Causes can range from ACLs that block UDP port 123 to router or NTP server misconfigurations.

Guidelines for Troubleshooting Network Services Issues

Here are some guidelines for troubleshooting common network services issues.

Troubleshooting Network Resolution and DNS Issues

Consider the following as you troubleshoot name resolution and DNS issues:

  • Determine what has caused the incorrect setting on the DNS server. If it is due to a manual error, you can modify the value suitably in the DNS server entries. However, if the error is on the DHCP server side, you need to check the DHCP server settings. In Cisco IOS, when you are using the IOS DHCP server feature in the DHCP pool mode, you need to specify this setting using the dns-server <server-ip-address> command.
  • Problems in establishing IP connectivity between the user's host and the DNS server can be frequently encountered in any network environment. The router can be configured with the IP addresses of the DNS servers, which would allow the router command's attempt to resolve names. For instance, a user of the router command-line interface (CLI) might issue a ping server15 command and rely on a DNS request to resolve server15 into a matching IP address. However, you need to issue the ip name-server dns1-address dns2-adddress global command to configure a router to use DNS for name resolution. The router also needs the ip domain-lookup global command, which is enabled by default.
  • Investigate whether or not the client is being misinformed about the destination's IP address.
    • Check the client's HOSTS file for inappropriate entries. In Windows, it is usually found in C:\Windows\System32\drivers\etc\. In Linux, it is usually found in /etc/.
    • Check to see if the client's resolver cache was corrupted with an inappropriate name-to-IP mapping (through a virus or other hacking).
      • In Windows, at a command prompt, issue the ipconfig /displaydns command and search for suspect entries. You can also enter ipconfig /flushdns first to clear dynamic entries, then issue ipconfig /displaydns to look for inappropriate persistent entries.
      • For Linux, the command you need to use will depend on your system. Common commands include:
        • sudo systemd-resolved --flush-caches
        • sudo systemctl restart dnsmasq.service
        • sudo /etc/init.d/dns-clean start (Ubuntu)
        • sudo systemctl restart nscd.service (RedHat)
        You can also simply reboot the client to flush its DNS resolver cache.
Troubleshooting DHCP Issues

When attempting to troubleshoot potential DHCP issues, remember the following guidelines:

  • You need to explicitly configure a router to act as a DHCP relay agent when the DHCP clients and DHCP server are on different subnets. Remember, by default, a router will not forward DHCPDISCOVER broadcast messages and other broadcast messages.
  • A DHCP pool can contain only a limited number of addresses. When the address limit is reached, new DHCP requests will be rejected.
  • Ensure that the DHCP server is not handing out the same IP address that has been statically assigned to a host on the network. Duplicate addresses can cause connectivity issues for both the host and the DHCP client.
  • When you have more than one DHCP server on the same network for redundancy, you need to ensure that the DHCP servers are configured such that they can communicate with each other. If for some reason the DHCP servers are not able to communicate with each other, the DHCP servers may end up assigning overlapping IP addresses to the DHCP clients.
Troubleshooting NAT Issues

While troubleshooting NAT issues, you need to follow certain guidelines:

  • Ensure that the configuration is correct. Use the show ip nat translation and debug ip nat commands.
  • Verify that packets are not denied from entering the NAT router by any inbound access lists.
  • Verify that on the NAT router the appropriate route is defined in the routing table when the packet is going from inside to outside.
  • Verify that all the necessary networks are permitted by the access list referenced by the NAT command.
  • Ensure that there are sufficient addresses available in the NAT pool. If not, this may cause a problem when the NAT is not configured for overloading.
  • Verify that there are appropriate definitions for the router interfaces as NAT inside or NAT outside.
Troubleshooting NTP Issues

When a device that is configured as an NTP client is not able to synchronize with an NTP server, you need to use these guidelines:

  • Use the ping command to test connectivity from the NTP client to the NTP server by using the correct destination IP address and source addresses.
  • Verify that there is no ACL between the NTP client and server that has been configured to deny NTP packets either by accident or on purpose. You need to use the show ip interface <interface-type> <interface-number> command to check if any ACLs exist on the interfaces and then use the show access list command to verify the ACL entries.
  • Use the show running-config | section ntp command to verify the NTP authentication configuration and verify that both the server and client are configured with the correct authentication key and key string.
  • Use the show ntp status command to verify the NTP server that is being used and use the ntp server <ip-address> prefer command to force a preferred NTP server.
  • Use the show process cpu command to verify that CPU load is not causing the NTP issue.
  • Use the show clock command to verify the clock that is set on the device. The NTP hierarchy uses stratum levels from 1 (best -- most reliable) to 15 (worst -- least reliable). When a device is synchronizing with another device that has a stratum of 15, the synchronization will fail.

Command Reference: Troubleshooting NTP

This section contains the commands you will use in the next lab.

Troubleshooting NTP

The following commands are used in the Troubleshooting NTP lab:

  • clock set <hh:mm:ss> <day> <month> <year>: Set the system software clock.
  • configure terminal: Enter global configuration mode from privileged EXEC mode.
  • enable: Enter privileged EXEC mode.
  • end: Ends and exits configuration mode.
  • exit: Exits one level in the menu structure.
  • ip name-server <ip-address>: Configures the name server for the device.
  • no ntp: Disables the NTP service on a device and removes the configuration.
  • [no] ntp master [<stratum>]: Configures the Cisco IOS software as an NTP master clock to which peers synchronize themselves. The no form removes the NTP server from the device.
  • ntp server <ip-address> [key <key-id>]: Allows the Cisco IOS software clock to be synchronized by an NTP time server.
  • ping <ip-address> | <host-name>: Sends an ICMP echo request to the specified address.
  • show clock [detail]: Displays the time and date from the system software clock.
  • show ntp associations [detail]: Shows the status of NTP associations..
  • show ntp status: Shows the status of NTP.
  • show running-config: Displays the active configuration file.

Command Reference: Troubleshooting DHCP

This section contains the commands you will use in the next lab.

Troubleshooting DHCP

The following commands are used in the Troubleshooting DHCP lab:

  • configure terminal: Enter global configuration mode from privileged EXEC mode.
  • enable: Enter privileged EXEC mode.
  • end: Ends and exits configuration mode.
  • exit: Exits one level in the menu structure.
  • interface <type> <number>: Changed from global configuration mode to interface configuration mode.
  • ip config: Used in NetSim to display the currently configured IP address, subnet mask, and default gateway on a workstation.
  • ip config /all: Used in NetSim to display the IP addresses and MAC address on a workstation.
  • ip dhcp excluded-address <ip-address> <last-ip-adddress>: Configures a range of excluded addresses.
  • ip dhcp pool <pool-name>: Creates an IP address pool.
  • lease [<days> [<hours> [<minutes>]] | infinite]: Configures a custom lease duration for DHCP assignments.
  • network <ip-address> <subnet-mask>: Enables a DHCP server for a particular network segment.
  • [no] service dhcp: Configures the router to respond to any DHCP requests. The no form disables the DHCP service and prevents a DHCP relay agent from forwarding DHCP requests.
  • ping <ip-address> | <host-name>: Sends an ICMP echo request to the specified address.
  • show ip dhcp binding: Displays active DHCP-assigned IP addresses.
  • show ip dhcp pool: Displays a basic summary of a configured DHCP pool.
  • show ip dhcp server statistics: Displays general operating statistics for a DHCP server process.
  • show ip interface [<type> <number>]: Displays IP information for an interface.
  • show running-config: Displays the active configuration file.

Troubleshooting DNS

The following commands are used in the Troubleshooting DNS labs:

  • configure terminal: Enter global configuration mode from privileged EXEC mode.
  • enable: Enter privileged EXEC mode.
  • end: Ends and exits configuration mode.
  • exit: Exits one level in the menu structure.
  • ip config /all: Used in NetSim to display the IP addresses and MAC address on a workstation.
  • ip config /dg <ip-address>: Used in NetSim to assign a default gateway IP address to a workstation interface.
  • ip config /dns <ip-address>: Used in NetSim to set the primary DNS server for a workstation.
  • ip config /ip <ip-address> <subnet-mask>: Used in NetSim to assign an IP address and subnet mask to a workstation.
  • ip dns server: Enables the DNS server on a router.
  • ip domain-lookup: Enables the DNS lookup feature.
  • ip host <host-name> <ip-address>: Configures a static host name-to-address mapping in the host cache of a device.
  • ip name-server <ip-address>: Configures the name server for a device.
  • no shutdown: Enables an interface.
  • ping <ip-address> | <host-name>: Sends an ICMP echo request to the specified address.
  • show hosts: Displays the default domain name, a list of name server hosts, and a cached list of host names and addresses.
  • show ip dhcp binding: Displays active DHCP-assigned IP addresses.
  • show ip dhcp pool: Displays a basic summary of a configured DHCP pool.
  • show ip interface brief: Displays a brief summary of interface status and configuration.
  • show ip protocols: Displays information about active routing protocols.
  • show ip route: Displays the IP routing table.
  • show running-config: Displays the active configuration file.
