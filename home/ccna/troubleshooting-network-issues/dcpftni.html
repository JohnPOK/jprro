Describe Common Processes for Troubleshooting Network Issues

Topic A

Troubleshooting Methodologies

One of the most important facets of successful troubleshooting is to use a systematic approach to identifying and resolving issues. By identifying common troubleshooting methodologies, you can adopt one to use as a standard framework for your troubleshooting activities.

Cisco Troubleshooting Model

Cisco offers an eight-step troubleshooting method to help you fix your network:

1. Define the problem.
2. Gather detailed information.
3. Consider probable cause for the failure.
4. Devise a plan to solve the problem.
5. Implement the plan.
6. Observe the results of the implementation.
7. Repeat the process if the plan does not resolve the problem.
8. Document the changes made to solve the problem.

Step 1: Define The Problem.

At the first indication of a potential issue, consider verifying that the network is working correctly by checking the documentation and network configuration. Documentation of network addresses is vital. You can use the Cisco IOS show and debug command outputs to check the topology and to test connectivity.

Be sure to record your ideas and actions at every step of the process.

Step 2: Gather Detailed Information

Communicate with the individuals who reported the problem to try and elicit information about what they were doing when the problem surfaced.

Network diagrams can help you resolve IP connectivity problems. Use them to compare the current network traffic with the expected working of the network.

One way to isolate a problem is to determine the path of a packet in the network and identify the point at which the network stops forwarding the packet. Use the documentation, network configuration, and the traceroute and ping commands. Test every hop of a packet from its source to destination to identify the exact issue.

Step 3: Consider Probable Cause For The Failure

Determine the cause of the problem and identify suitable actions that can be performed to resolve the problem.

Problems in the network can be due to connectivity issues between a source and destination. Some of the common causes of network problems are Physical layer issues, first-hop Layer 3 connectivity issues, Layer 3 IP connectivity issues along the path of the packet, and name resolution issues.

If you are successful at a certain level of the OSI model, you can probably assume that the lower levels are operating correctly. For example, if you ping a router interface, it is unlikely that there is a cable issue.

Step 4: Devise A Plan To Solve The Problem.

When you have identified a cause, develop a resolution that will address the cause.

Some of the commonly used approaches to resolve connectivity issues include:

  • Verifying that the entire route (from source to destination) is available with the router.
  • Fixing software problems or other configuration problems in a router.
  • Fixing physical connectivity problems.
  • Checking for resource availability problems, such as lack of memory on a router.
  • Identifying routing loops and routing-protocol-related problems.
  • Identifying whether traffic is blocked by firewalls or packet filters.

Symptoms can be misleading. Many problems manifest themselves as client connectivity issues. Make sure you check things like ACL configuration even if you don't think the problem is a security issue.

Part of correcting the problem also includes determining if you should resolve it yourself or escalate the issue. If you are unable to resolve the network issue, you need to escalate it to the next level or to Cisco System's Technical Assistance Center (TAC). The network diagrams, output of show and debug commands, and key information you gathered while troubleshooting will be useful in issue resolution.

Step 5: Implement The Plan.

As you implement your plan, try to change only one thing at a time. That way, if the change does not produce the expected results, you can undo the change and try another approach. Recording what you changed and how it affected the network is extremely important.

Step 6: Observe The Results Of The Implementation.

Verify thet your plan addressed the problem and did not introduce additional issues.

After you have identified and resolved any network problems, it is important to verify the operation of the network. You can do this by checking the flow of packets along their respective paths in the network topology. Executing commands such as show ip, show interface, and show running-config on a Cisco router can help to determine the current status of the network. The debug command enables you to monitor the network operation over a period of time. In addition, checking the log messages generated by the debug command can help you determine if the network is functioning as expected.

Step 7: Repeat The Process If The Plan Does Not Resolve The Problem.

Sometimes troubleshooting is an iterative process. If your plan does not provide the intended results, back out of the changes and return to Step 4 to develop and implement a new plan.

Step 8: Document The Changes Made To Solve The Problem.

Use the information you recorded as you worked through the troubleshooting process to document the reported problem, its cause, your plan of attack, and the results of implementing the plan. This information can be vitally important if the issue arises again in the future.

Log Files and Troubleshooting

Log files generated by Cisco devices can enable you to identify the causes of issues, so that you can take suitable measures to troubleshoot the issues faced. The show logging command allows you to view the state of system logging (syslog) and the contents of the standard system logging buffer.

Routers and switches issue log messages in response to different events. For example, if an interface fails, the device creates log messages. With default settings in place, Cisco IOS sends these messages to the console port. You can also configure Cisco IOS to handle log messages by applying filters and specifying the delivery destinations for the log messages.

Most devices display messages and errors in real-time to their console. If you wish to observe an activity step by step, you can also turn on debugging. Common use cases for debugging include observing:

  • Pings or general IP traffic
  • NAT translations
  • Routing protocol handshakes, adjacency establishment, and route updates
  • Dialup or VPN connection establishment and teardown

You should also be aware that if you are managing a Cisco device from a remote console, you might not be seeing all of the error messages. It is possible to set the amount of messaging you would like to see on your console.

  • Note: Console messages and debug output are not sent to vty terminal (Telnet or SSH) sessions by default. Use the terminal monitor command within the vty session to display console messages.

Command Reference: Debugging

This section contains the commands you will use in the next lab.

Debugging

The following commands are used in the Basic Debugging lab:

  • debug ip icmp: Displays IP diagnostic information on ICMP transactions.
  • debug ip rip: Displays IP diagnostic information on RIP routing transactions.
  • enable: Enters privileged EXEC mode.
  • no debug all: Turns off all diagnostic output.
  • ping <ip-address>: Sends an Internet Control Message Protocol (ICMP) echo request to the specified address.
  • show debugging: Determins which type of debugging is enabled.
