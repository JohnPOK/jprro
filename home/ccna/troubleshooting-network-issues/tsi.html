Topic I

Troubleshoot Security Issues

With today's security concerns affecting practically all users on all networks, the ability to troubleshoot network issues is an essential tool for any network administrator.

Common Security Issues

Some common security issues you might encounter include:

  • Issues with port security configurations.
  • Issues related to device hardening.
  • Issues related to console access.
  • Issues with ACLs.

Port Security Issues

Issues related to port security include:

  • Port security violations. When you enable port security, each interface has a default or configured number of MAC addresses that can be secured. The number of MAC addresses that can be secured per port must be determined and the interface must be configured with the same number of addresses. When port security is applied under anticipated operating conditions and with proper configuration, it will continue to work normally until a port security violation occurs.
    • Causes of port security violation include virus attacks, hostile nodes, or accidental reconfiguration of hosts, and can result in an end host sending out packets with more MAC addresses than expected. This will result in port security violation. The system will log an error message indicating that a violation has occurred. The error message will also include the MAC address of the end host and the port on which the violation occurred. The system will send an alert or a trap if the SNMP traps for port security are enabled.
    • Depending on the violation mode, the port can be err-disabled or set to shutdown mode, or the packets from the unsecure addresses can be dropped in the software or set to restrict mode. However, to ensure that the CPU is not loaded when such an event occurs, you need to set the violation mode to shutdown. You can ensure an automatic recovery from the err-disabled state by configuring errdisable recovery and timeout.
  • Configuration management.
    • The secure trunk port can be configured with the static or sticky secure addresses. However, if you change the set of allowed VLANs on the trunk port, the configured addresses located on certain VLANs will be erased by the software. Then, the system will log an error message that identifies the VLANs from which the VLAN-based maximum and static or sticky secure addresses are erased by the software.
    • You can convert a port configured with port security and static or sticky secure addresses to a router port or hotswap it out of the system. This will make the configuration on the port ineffective in the system. The addresses that are secured on the port can be secured on other ports.
    • To ensure that the configuration is restored when the port is either converted back to a Layer 2 switchport or hotswapped into the system, the software will retain the the configuration. When such a port is made available, the software ensures that the configured static or sticky secure addresses are not secured on other ports. If the software observes that the address has been used on a different port than where it was inserted, it will erase the address.

Device Hardening

Performing device hardening on network computers and devices is meant to keep them safe from unauthorized access, but you might run into cases where a PC, switch or router has been configured so tightly that it can't provide the services that users need. Some facets of device hardening that you might need to troubleshoot include:

  • Console access issues. These can be a matter of not enough security (meaning unauthorized users can access the console of a Cisco switch or router), or you might encounter situations where IT personnel who should be able to access a console can't do so.
  • ACL issues. Again, this might be a situation where an ACL is too restrictive (users who should be able to access a resource can't) or not restrictive enough (unauthorized users can access resources that they should not be able to access).

Console Access

The default method for accessing Cisco routers and switches is by using the console port (sometimes referred to as the management port). However, most network administrators want the convenience of being able to gain access to a router or switch console from a remote or centralized location. Here are some tips for restricting console access:

  • Decide if you want or need to allow direct connection access to network devices (via a serial port). This decision migth depend on many factors, including organizational security policies.
  • Make sure that you use the Cisco rollover cable to connect to the console port. This cable is light blue and provide a unique pinout.
  • If you decide to prohibit direct connection access and allow only remote access, you might think that disabling the physical console port is the best approach. It would not affect remote access to the device, since that occurs through the existing network media. However, in some troubleshooting scenarios you will want to be able to make a direct connection to a device. A more appropriate action is to physically secure the device in a locked cabinet or room. This will prevent "just anyone" from making a direct connection, but still enables you to do so when the need arises.
  • Whether for direct access or remote access, you definitely want to require user name/password credentials, and you want to be sure to set the appropriate level of encryption on the password so that it is not easily compromised.

ACL Configuration Issues

Even before you attempt to troubleshoot ACL issues, you need to focus on these four key points:

  • Determine if there is an ACL problem: If possible, remove the access list and test.
  • Test the interfrace where the ACL is applied, and determine the direction of data flow: You may be tempted to verify the access list first, but it is also important for you to verify the direction that data is flowing, and ensure that the sources and destinations are being correctly interpreted. It might be possible that the applied interface is not on the path the traffic is routed through, or that the ACL has been applied in the wrong direction, such as incoming when it should be outgoing.
  • Check the entries in the access list: Verify that the addresses and port numbers have been entered correctly.
  • Review the order of operations: Remember that ACLs are processed top-down, and the first match (permit or deny) is executed. It could be possible that the traffic is matching an earlier statement. You can check this by temporarily moving the line in question to the top of the list.

Some connectivity issues can result from misconfigured ACLs. The basic premise comes down to two possible issues:

  • Traffic that you want to block is being allowed.
  • Traffic that you want to allow is being blocked.

The following table describes common problems, along with possible causes and solutions.

Problem Possible Cause Solution
All packets are getting blocked:
  • The access list might be empty.
  • A deny filter may be too broad.
  • A deny filter may be too high in the access list order.
  • No existing permit filters can match the packets.
  • You created rules to explicitly deny various types of traffic, then forgot to include a "permit any" rule at the end.
  • Use the no ip access-group or the no ipv6 traffic-filter command to remove the access list from the interface.
  • Use the no ip access-list command for IP version 4 (IPv4)-ACLs or no ipv6 access-list command for IPv6, and use the no deny command to delete the deny filter.
  • Delete the access-list and recreate it.
  • Use the ip access-list command for IPv4-ACLs or ipv6 access-list for IPv6, and use the permit command to add the permit filters.
  • Make sure the last statement in your ACL permits all traffic.
  • For a standard ACL enter access-list <number> permit any. For an extended ACL enter access-list <number> permit any any.
No packets are getting blocked:
  • A permit filter may be too broad.
  • The permit filter may be too high in the access list order.
  • The filter might be applied to the wrong interface.
  • The filter might be applied to the right interface, but in the wrong direction.
  • Use the no ip access-list command for IP version 4 (IPv4)-ACLs or no ipv6 access-list command for IPv6, and use the no deny command to delete the deny filter.
  • Delete the access list and recreate it.
  • Draw a diagram on paper with arrows indicating the flow of traffic. Ensure that the interface you applied the ACL to is in the path of the data flow.
  • Using a traffic-flow diagram, verify that the direction of the ACL on the interface (in, out) corresponds to the direction of the traffic flow. Pay special attention to source and destination IP addresses and ports.
PortChannel not working with ACL:
  • The ACL may not have been applied to all interfaces in the PortChannel.
  • Use the show port-channel database command to determine the interfaces that are part of the PortChannel. Then, you need to use the ip access-group or the ipv6 traffic-filter command to add the ACL to all interfaces in the PortChannel.
Cannot remotely connect to switch:
  • There may be an incorrect ACL on the mgmt0 interface.
  • Locally connect to the console port and delete the ACL, using the no ip access-group or the no ipv6 traffic-filter command.

Guidelines for Troubleshooting Security Issues

Here are some guidelines for troubleshooting security issues.

Troubleshooting Port Security

To resolve problems related to port security, you need to apply the solutions described in the following table.

Problem Cause/Solution
  • Secure addresses do not appear as static entries in the output of show mac address-table static command.
  • Cause: The configuration of the secure addresses is not complete.
  • Solution: Check the address and make it secure by associating the VLANs.
  • The previously configured static or sticky secure addresses on VLANs are missing.
  • Cause: If you change the set of allowed VLANs on the trunk port, the configured static or sticky secure addresses located on certain VLANs are erased by the software. You can convert a port configured with port security and static/sticky secure addresses to a router port. Also, you can hotswap it out of the system. If this happens, the configuration on the port is ineffective in the system. This allows the addresses that are secured on the port to be secured on other ports.
  • Solution: You need to configure a secure trunk port with static or sticky secure addresses.
  • There is a port security violation and the systems have logged the error message Security violation occured. The systems will send a trap if you have enabled SNMP traps for port security. Based on the violation mode, either the port will be error-disabled (shutdown mode), or the packets from the unsecure addresses may be dropped in the software (restrict mode).
  • Cause: Each interface has a default or configured number of MAC addresses that can be secured when port security is enabled. You need to determine the number of MAC addresses that can be secured per port and configure that many number of addresses on the interface. When port security is properly configured under anticipated operating conditions, port security will continue to work normally. However, virus attacks, hostile workstations, or accidentally reconfigured hosts, might cause end hosts to send out packets with more than the expected number of MAC addresses. This will result in a port security violation and the resultant system error messages will appear based on the configuration.
  • Solution: To ensure that the CPU is not loaded when such an event occur, you need to set the violation mode to shutdown. You need to configure errdisable recovery and timeout to ensure an automatic recovery from the err-disable state.
Troubleshooting Console Access

One of the ways to implement device hardening is to restrict the number of users who have permission to remotely access the Cisco devices. The default out-of-the-box method of accessing Cisco routers and switches is using the console port. Some of the factors that you need to consider when troubleshooting console access include:

  • Ensure that the correct COM port has been selected in the terminal program. Mostly, multiple COM ports are displayed in the terminal program, but the last one listed is usually the correct one to use. If it is not, you need to try a different one. This is really a trial-and-error process to determine the correct COM port.
  • Ensure that the terminal program settings are configured correctly. Remember that the Cisco devices use the following default values: 9600 baud, 8 data bits, 1 stop bit, no parity.
  • Determine if a line password is used to authenticate access to the console. When a line password is being used, the login command needs to be configured as well. Remember that the login command and a line password are not configured by default.
  • Ensure that the local user name and password are used to authenticate access to the console. When local authentication is being used, a user name and password needs to be defined in the local database, and the login local command is also required.
  • Determine if a AAA server is used to authenticate access to the console. When AAA authentication is being used, you need to define a method list with the login authentication { default | list-name } command line console in configuration mode.
Verifying the Connecting Cables and Drivers

You need to verify that the correct cable and drivers are being used to connect to the console port. Check your device's documentation to determine what is needed. Newer devices may be using a mini USB port as the console port (drivers required on PC), while older devices may still be using the serial to RJ45 (rollover) cable.

Troubleshooting ACL Issues

You can begin troubleshooting ACLs by verifying that:

  • The access list has been applied to the interface and in the proper direction.
  • The access list is not empty.
  • The order of the rules in the access list is correct.
  • If there are several deny statements in an ACL, make sure that a "permit any" statement appears at the end of the list.
