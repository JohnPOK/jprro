Topic F

Perform Network Management

The task of managing the network and its resources is a major responsibility. Many tools exist to assist in network management, and your ability to select the tool that best fits your needs will reduce downtime, costs, and frustration levels.

Network Management Basics

Network Management is the practice of administering computer networks. Common goals of network management include:

  • Ensure the network supports the organization's business requirements.
  • Deliver high-performing services to users that depend on the network to do their jobs.
  • Save time and money by automating repetitive or time-consuming tasks.
  • Improve efficiency and security by standardizing policies, procedures, configurations, and equipment.
  • Reduce costly downtime through automated troubleshooting and fault analysis.
  • Improve systems performance through proactive monitoring, maintenance, and capacity forecasting.

SNMP

The Simple Network Management Protocol (SNMP) is a set of network management standards for IP-based internetworks. It includes a protocol, a database-structure specification, and a set of management data objects.

SNMP implementations typically consist of a management application running on one or more network management systems (NMSs), and agent applications, usually executing in firmware on various network devices.

SNMP has three major standard revisions: SNMPv1, SNMPv2, and SNMPv3. The Cisco devices support all three versions, with version 3 being recommended, a standard Management Information Base (MIB-II) object, along with a special set of enterprise MIB objects.

SNMPv1 was initially developed as an interem management protocol, and major vendors incorporated SNMP modules into their network systems and components. This led to the enhancement of the protocol and to the development of SNMP version2 (SNMPv2). SNMPv3 was introduced to meet the need for better security because SNMPv2 had no security enhancements. It supports secure authentication and encryption of all SNMP messages.

Agents

An SNMP agent is a software component running on a managed device. It maintains data about the device and provides that data to management systems. The device information is stored in MIBs.

The Management Information Base (MIB) defines variables, which enable the management software to monitor or control the network device. Initially, the MIB defines each variable as an object ID (OID). Then, the MIB organizes the OIDs based on Request for Comments (RFC) standards into a hierarchy of OIDs, usually in a tree structure.

Syslog

Syslog (RFC 5424) is a standard for message logging. Because so many different network devices use the Syslog standard, you can compile log information for disparate devices in a single, centralized location.

Every Syslog message includes a priority value ranging from 0 to 191 included in angle brackets (<>). This value is calculated according to this formula:
Priority = Facility*8 + Severity Level

Where Facility represents the machine process that created the event, and Severity Level is one of the levels defined in the standard.

By default, Cisco network devices send system messages and debug output to a local logging process stored inside the device. You can manually configure the logging process to allow the messages to be sent to different destinations according to your needs. The destination for the messages can be a syslog server, or, the messages can be sent to an internal buffer for storage, which can be viewed through the device CLI later. You can also filter the type of system messages that need to be sent to any syslog server or other destinations.

QoS

QoS deals with the ability of a network to offer improved service to specific types of network traffic. It is defined by a set of parameters that controls the service provided. QoS prioritizes traffic based on importance and uses congestion avoidance methods.

The most common QoS parameters are bandwidth, latency, jitter, and packet loss. QoS aims at providing high bandwidth, less latency and jitter, and minimal data loss while making sure that prioritizing one type of network traffic does not affect the other types. The following table describes these QoS parameters.

Qos Parameter Description
Bandwidth The average number of bits of data that can be transmitted from source to destination in one second.
Latency The time difference between transmision of a signal and its receipt. Also called lag or delay.
Jitter The variability over time in latency between sequentially transmitted packets.
Packet Loss The number of packets lost or damaged during transmission.

An optimum usage of bandwidth is critical for multimedia applications. Low bandwidth may result in bad quality transmission leading to packet dropouts or data loss. Latency is inevitable when packets are held up in queues or when packets are routed through less congested routes that are long. However, latency can be minimized by increasing the network bandwidth, fragmenting data packets, or prioritizing data on a network. Jitter may be caused by network congestion, improper queuing, or configuration errors. A very low jitter is important for voice and video transmissions. Packet loss may be due to full buffers at the destination. This can result in destinations requesting retransmission of data.

Two approaches to applying QoS are Integrated Services (IntServ) and Differentiated Services (DiffServ).

  • IntServ reserves bandwidth between two nodes before communication even begins.
  • DiffServ employs packet prioritization to ensure higher priority traffic is favored.

Per-Hop Behavior

Per-hop behavior (PHB) is externally observable forwarding behavior such as packet scheduling, queuing, policing, or shaping behavior that is applied at a DiffServ-compliant node to a DiffServ Behavior Aggregate. PHBs define the packet-forwarding properties for a certain class of traffic.

  • Note: For more information about DiffServ, visit https://www.cisco.com/c/en/us/td/docs/ios-xml/ios/qos_dfsrv/configuration/15-mt/qos-dfsrv-15-mt-book/qos-dfsrv.pdf.

Classification

Classification is the process of differentiating network traffic types to implement Qos. It is an identification step that examines the fields in packets to classify network traffic.

Classification can be applied in two ways:

  • On a per-hop basis: Packet is identified at a router but its class is not marked. Class information is not passed on to other routers.
  • On a network-wide basis: Packet is identified at a router and marked with IP Precedence (IPP) or Differentiated Services Code Point (DSCP) values for network-wide identification. Applied when Priority Queuing (PQ) and Custom Queuing (CQ) methods are used.

Some of the common methods to perform classification are:

  • Matching Class of Service (CoS), IPP, or DSCP values or access control lists (ACLs).
  • Policy-based routing.
  • Committed Access Rate (CAR).
  • Network-Based Application Recognition (NBAR).

Marking

Marking is the next step after classification. Marking happens at Layer 2 and Layer 3 of the OSI model. At Layer 2, CoS is identified, and at Layer 3, the IPP and DSCP values are identified.

A value is set in an IP packet header depending on the matched class. The value can be an IPP, CoS, or DSCP value, which identifies the traffic class. The newly marked values determine the QoS action on the packet.

Earlier Internet Protocol version 4 (IPv4) packets were marked with IPP and CoS values. Currently IPv4 and IPv6 use a 6-bit DSCP field to mark packets. In IPv6, the DSCP field is also called the traffic class byte. When 802.1Q headers are used in VLAN trunk links, a 3-bit CoS or Priority Code Point (PCP) field shows the markings in an Ethernet frame header.

Queuing

Queuing is the method of assigning higher priority to some queues so that packets in higher priority queues are forwarded first. Cisco devices such as routers and switches use different types of prioritization techniques while forwarding packets from queues:

  • Round-robin: Scheduler forwards a specific number of packets or bytes from one queue and moves on to the next queue until it cycles back to the first queue.
  • Weighted round-robin: Scheduler cycles through queues but forwards a different number of packets or bytes in each queue based on the priority levels assigned to the queues. Class-Based Weighted Fair Queuing (CBWFQ) is one tool that uses this method.
  • Low-Latency Queuing (LLQ): Scheduler places voice, video, and data packets into separate queues and assigns different priority levels for each queue.

Voice Prioritization

Considerations for prioritizing voice include:

  • Interactive voice applications like IP phone calls need less latency, jitter, and packet loss for good quality audio transmissions.
  • Voice over IP (VoIP) calls require lesser bandwidth than data applications.
  • Cisco guidelines provide permissible limits on the latency, jitter, and packet loss QoS parameters to ensure good quality audio transmissions.
  • Priority queuing with LLQ is the most preferred prioritization mechanism for interactive voice transmissions.
  • CBWFQ is the most preferred queuing method for non-interactive voice transmissions.
  • The following table lists the Cisco recommended limits for different QoS parameters while prioritizing voice.
    Qos Parameter Recommended Limits
    Latency 150 ms or less for one-way flow
    Jitter 30 ms or less
    Packet Loss 1% or less

Video Prioritization

Considerations for prioritizing video include:

  • Video applications need less latency, jitter, and packet loss for good quality transmissions. They also require high bandwidths.
  • You need careful configuration of QoS parameters to ensure good user experience.
  • Cisco guidelines provide permissible limits on the bandwidth, latency, jitter, and packet loss QoS parameters to ensure good quality video.
  • Priority queuing with LLQ is the most preferred prioritization mechanism for interactive video transmissions like video conferencing.
  • Specifying bandwidth rates for each priority queue helps to avoid packet loss.
  • The table lists the Cisco recommended limits for different QoS parameters while prioritizing video.
    Qos Parameter Recommended Limits
    Bandwidth 384 Kbps to more than 20Mbps.
    Latency 200 to 400 ms or less for one-way flow
    Jitter 30 to 50 ms or less
    Packet Loss 0.1% to 1%

Data Prioritization

Considerations for prioritizing data include:

  • Data applications can tolerate slightly higher latency, jitter, and packet loss than voice and video applications.
  • Interactive data applications require high bandwidths to provide fast and reliable services to users.
  • Data transmissions typically occur in bursts. For example, a user selecting an option on a web page that results in large downloads of data from the web server.
  • In an enterprise, traffic involving data are classified based on importance to ensure that business-critical applications do not suffer data loss.
  • CBWFQ is commonly used as the scheduling tool for prioritizing data applications.
  • Call Admission Control (CAC) tools are used to prevent accumulation of data packets when audio and video transmissions are given higher priority.
  • When prioritizing traffic of different types, care should be taken to ensure that data packets are not delayed in queues for very long time periods.

Traffic Congestion

The process of managing queues to reduce congestion at networking devices is called congestion management. Networking devices such as routers maintain queues at their outgoing interfaces. Packets are stored in queues until an outgoing interface is available for transmission.

Packets in queues are marked with class information that determines their priority. When multiple queues are used, a router uses a scheduler function to determine which packet has to be selected from which queue for transmission. Schedulers use prioritization techniques to prioritize queues and manage congestion.

Traffic Policing

Traffic Policing is the process of creating a policer in a policy map. A policer is a command within a policy map that defines the bandwidth limits for network traffic and outlines the actions to be taken depending on whether a packet conforms to the limits or not. The actions defined in the policer are implemented using a marker. The policer uses the length of the IP payload, which is the length of the IP packet header, to perform the required checks.

Policers drop or re-mark traffic without introducing delay or jitter.

Traffic Shaping

Traffic Shaping is a mechanism to control traffic flow at an outgoing interface. A shaper is a tool that queues packets at an outgoing interface. This helps to reduce the speed of traffic flow to a rate that can be handled by the destination. Shapers use shaping rates to schedule transmission of packets from the output queues. A shaper ensures that the transmission rate of packets does not exceed the specified shaping rate.

Shapers might delay traffic, which can introduce jitter.

A congestion management tool such as CBWFQ or LLQ scheduling is used to manage congestion at the output queues maintained by a shaper.
